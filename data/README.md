# WEMEET 학습 데이터 자산 정리

> 목적: 현재 저장소에 포함된 학습자료 덤프를 포트폴리오에서 어떻게 설명할지, 그리고 앞으로 어떤 구조로 확장할지를 정리한다.

---

## 1) 현재 확보된 데이터

### 파일 목록
- `data/questions.json`

### 현재 상태 요약
- 객관식 문제 리스트(JSON 배열)
- 현재 커밋 기준 **60개 문항** 포함
- 각 문항은 아래 필드를 가짐
  - `question`
  - `options`
  - `answer`

즉, 현재 데이터는 “서비스 검증용 시드 문제셋” 역할을 하며,
초기 학습 엔진/퀴즈 API/오답 저장 흐름을 검증하기에 충분한 최소 자산입니다.

---

## 2) 현재 데이터의 의미

이 데이터는 단순 샘플 파일이 아니라,
포트폴리오에서는 아래처럼 설명할 수 있습니다.

- 학습 플랫폼 MVP를 검증하기 위한 **초기 문제 자산을 직접 준비**했다.
- 문제/선택지/정답 구조를 JSON 형태로 먼저 통일해,
  이후 DB 적재와 API 제공이 가능하도록 **데이터 표준화의 출발점**을 만들었다.
- 향후 AWS SAA, 전산 7급, 정보보안기사 등으로 확장할 때,
  동일한 포맷으로 문제셋을 추가할 수 있는 최소 스키마를 확보했다.

즉, “자료를 모을 예정”이 아니라,
**이미 자료를 덤프 형태로 확보했고 이를 서비스 데이터 파이프라인으로 전환하는 단계**라고 설명하는 것이 좋습니다.

---

## 3) 현재 스키마

### 현재 JSON 예시 구조
```json
{
  "question": "문제 본문",
  "options": ["선지1", "선지2", "선지3", "선지4"],
  "answer": 2
}
```

### 장점
- 구현이 단순해 빠르게 퀴즈 기능 검증 가능
- 프론트/백엔드 연동이 쉬움
- 초기 MVP 구축 속도가 빠름

### 한계
현재 구조만으로는 아래가 부족합니다.
- 시험 종류(`exam`)
- 과목/도메인(`domain`)
- 세부 개념(`concept`)
- 난이도(`difficulty`)
- 해설(`explanation`)
- 키워드(`keywords`)
- 연관 문제 링크(`related_ids`)
- 출처/정리일(`source`, `collected_at`)

따라서 지금 데이터는 **1차 덤프/시드 데이터**로 두고,
다음 단계에서 정규화가 필요합니다.

---

## 4) 추천 확장 스키마

향후에는 아래와 같은 구조로 확장하는 것이 좋습니다.

```json
{
  "id": "aws-saa-001",
  "exam": "AWS_SAA",
  "domain": "Storage",
  "concepts": ["S3", "Lifecycle Policy"],
  "question": "문제 본문",
  "options": ["선지1", "선지2", "선지3", "선지4"],
  "answer": 2,
  "explanation": "정답 해설",
  "keywords": ["s3", "lifecycle", "cost optimization"],
  "difficulty": "medium",
  "related_ids": ["aws-saa-014", "aws-saa-103"],
  "source": "personal_dump",
  "collected_at": "2026-03-20"
}
```

이 구조가 되면 다음이 가능해집니다.
- 시험별 문제 필터링
- 개념 그래프 연결
- 오답 기반 취약 개념 분석
- AI 요약/추천 입력값 생성
- 관리자 데이터 검수 도구 구축

---

## 5) 포트폴리오에서 강조할 포인트

### 좋은 표현
- 학습 플랫폼 MVP 검증을 위해 초기 문제 덤프를 직접 정리하고,
  이를 JSON 기반 시드 데이터로 구조화했습니다.
- 단순 UI 구현이 아니라, 이후 확장 가능한 문제 데이터 스키마와 적재 흐름까지 고려해 설계했습니다.
- 향후 AWS SAA, 전산 7급, 정보보안기사 등 다양한 시험 데이터를 동일 스키마로 통합할 수 있도록 데이터 표준화 방향을 수립했습니다.

### 피해야 할 표현
- 데이터는 나중에 넣을 예정입니다.
- 일단 샘플 문제만 조금 넣었습니다.
- 아직 구조화된 데이터는 없습니다.

사용자는 이미 덤프 파일을 준비했기 때문에,
이 프로젝트는 **실제 데이터 확보 및 구조화 단계에 진입한 상태**로 설명하는 편이 훨씬 강합니다.

---

## 6) 다음 작업 우선순위

### Step 1 — 시드 데이터 정리
- `questions.json`에 `id`, `exam`, `domain` 필드 추가
- 정답 외에 `explanation` 필드 추가
- 문제별 키워드 태깅

### Step 2 — 도메인 확장
- AWS SAA 전용 문제셋 분리
- 정보보안기사 문제셋 분리
- 전산 7급 문제셋 분리

### Step 3 — 데이터 파이프라인화
- JSON → DB 적재 스크립트 작성
- 관리자 검수용 입력 포맷 정의
- 중복 문항 탐지 기준 수립

### Step 4 — AI/추천 연결
- 오답 기록과 문제 메타데이터 연결
- 취약 개념 추출 로직 연결
- 관련 문제 추천 규칙 정의

---

## 7) 이력서/자소서에 녹이는 한 줄

- 학습 플랫폼 검증을 위해 초기 문제 덤프를 직접 구조화하고, 이를 서비스형 데이터 자산으로 전환하는 데이터 표준화 작업을 수행했습니다.
- 문제·정답·선택지를 JSON 시드 데이터로 정리한 뒤, 향후 시험별/개념별 확장이 가능하도록 정규화 스키마를 설계했습니다.

이 문장은 단순한 “문제 넣기”가 아니라,
**데이터 엔지니어링 감각 + 서비스 설계 감각**을 함께 보여줍니다.
